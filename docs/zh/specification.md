# DCL 演算法規格書

**方向約束鎖定（Directional Constraint Locking）**
版本 0.2 — 2026-03-25

---

## 1. 概述

方向約束鎖定（DCL）是一種以空間移動驅動的多維篩選演算法。使用者在二維畫布上沿 8 個方向導航；每個方向對應一個資料維度。往某方向移動會鎖定該維度的值，鎖定的約束持續累積，逐步縮小候選池。

透過 FIFO 解鎖機制，演算法保證導航永遠不會陷入死路。

---

## 2. 形式化定義

### 2.1 符號表

| 符號 | 定義 |
|------|------|
| C | 所有卡片的集合，\|C\| = n |
| D | {↖, ↑, ↗, ←, →, ↙, ↓, ↘}，共 8 個方向 |
| A(c, d) | 屬性函數：卡片 c 在方向 d 上的值。A : C × D → {1…8} |
| L | 當前鎖定集合，L ⊆ D × {1…8} |
| Q | 鎖定順序佇列（FIFO），記錄方向被鎖定的先後順序 |
| c_t | 時刻 t 的當前卡片 |

### 2.2 排列約束

對所有卡片 c ∈ C，函數 A(c, ·) : D → {1…8} 為**雙射（排列）**。

即每張卡片的 8 個方向值是 {1, 2, 3, 4, 5, 6, 7, 8} 的重新排列，不重複。

**意涵**：同一張卡片不可能在兩個方向持有相同的值。這確保了鎖定約束時的最大區分度。

### 2.3 候選池

```
P(L, c_t) = { c ∈ C | c ≠ c_t ∧ ∀(d, v) ∈ L : A(c, d) = v }
```

候選池是所有滿足每個鎖定約束的卡片集合（排除當前卡片）。

### 2.4 導航函數：nav(d)

給定方向 d，導航步驟如下：

```
1. 鎖定 — 若 d 尚未被鎖定：
     L ← L ∪ {(d, A(c_t, d))}
     Q.enqueue(d)

2. 查詢 — 計算候選池：
     P ← P(L, c_t)

3. FIFO 解鎖 — 若 P = ∅，釋放最早的鎖定直到有候選：
     while P = ∅ 且 Q ≠ ∅：
       d' ← Q.dequeue()        // 最早鎖定的方向
       L  ← L \ {(d', ·)}      // 移除該約束
       P  ← P(L, c_t)          // 重新計算

4. 循環 — 從池中選取下一張卡片：
     c_t ← P[k mod |P|]
     k   ← k + 1
```

**已鎖定方向的行為**：當 d ∈ L（方向已鎖定），步驟 1 被跳過 — 不新增約束。函數直接以現有鎖定集進入步驟 2，效果等同於在同一候選池中繼續循環。

## 3. 性質

### 3.1 單調累積

在導航過程中 |L| 單調不遞減，除非 FIFO 解鎖觸發。每次 `nav(d)` 呼叫要麼增加一個約束，要麼保持 L 不變。

### 3.2 最大鎖定數

|L| ≤ |D| = 8。最多只能鎖定 8 個不同的方向維度。

### 3.3 無死路保證

**定理**：對任意卡片集合 C（|C| > 1）和任意當前卡片 c_t，導航函數 `nav(d)` 總能產生非空候選池。

**證明概要**：
- 最壞情況下，FIFO 解鎖移除所有約束，得到 L = ∅。
- 當 L = ∅ 時，P(∅, c_t) = C \ {c_t}。
- 因為 |C| > 1，所以 |P(∅, c_t)| ≥ 1。
- 因此步驟 3 的 while 迴圈必定在 P ≠ ∅ 時終止。 ∎

### 3.4 循環導航

在固定鎖定集 L 下，候選池 P(L, c_t) 是有限的。在同方向持續導航會依序遍歷 P，最終回到第一個候選。

循環長度等於 |P(L, c_t)|。

### 3.5 排列唯一性

因為 A(c, ·) 對每張卡片都是雙射，一張卡片在每個方向最多匹配一個值。這避免了約束匹配的歧義，並確保鎖定集 L 的良定義性（每個方向最多出現一次）。

## 4. 複雜度分析

### 4.1 時間複雜度

| 操作 | 複雜度 |
|------|--------|
| 鎖定一個方向 | O(1) |
| 計算 P(L, c_t) | O(n · \|L\|)，其中 n = \|C\| |
| FIFO 解鎖（最壞情況） | O(\|L\| · n · \|L\|) = O(n · \|L\|²) |
| 單次 nav(d) 呼叫 | 最壞 O(n · \|L\|²)，典型 O(n · \|L\|) |

由於 |L| ≤ 8，每次導航步驟的實際複雜度為 **O(n)** — 與卡片數量線性相關。

### 4.2 空間複雜度

| 組件 | 空間 |
|------|------|
| 卡片儲存 | O(n · \|D\|) = O(8n) = O(n) |
| 鎖定狀態 | O(\|D\|) = O(1) |
| 鎖定佇列 | O(\|D\|) = O(1) |
| 循環計數器 | O(造訪過的不同鎖定狀態數) |

### 4.3 優化方向

對大型卡片集，O(n) 的池計算可以改進：

- **按方向-值對建索引**：預計算 `Index[d][v] = {A(c,d) = v 的卡片}`。池計算變成鎖定維度上的集合交集：O(最小桶大小 · |L|)。
- **位圖交集**：將每個 `Index[d][v]` 表示為位元集。Pool = 所有鎖定位元集的 AND 運算。O(n/64 · |L|)。

## 5. 資料結構

### 5.1 卡片節點

```json
{
  "id": "card_001",
  "attrs": {
    "right": 8,
    "left": 3,
    "up": 7,
    "down": 4,
    "rightUp": 6,
    "leftUp": 2,
    "rightDown": 5,
    "leftDown": 1
  }
}
```

`attrs` 物件將 8 個方向各映射到 {1…8} 中的一個唯一整數。

### 5.2 導航狀態

```json
{
  "current": "card_001",
  "lockMap": { "right": 8, "up": 7 },
  "lockOrder": ["right", "up"],
  "counter": { "right=8;up=7": 2 },
  "history": ["card_003", "card_007", "card_001"]
}
```

| 欄位 | 用途 |
|------|------|
| `current` | 當前活動卡片 |
| `lockMap` | 已鎖定的方向→值對應 |
| `lockOrder` | 鎖定順序的 FIFO 佇列 |
| `counter` | 每個唯一鎖定狀態的循環位置 |
| `history` | 近期導航軌跡 |

| 方法 | 相似之處 | 不同之處 |
|------|----------|----------|
| Faceted Search | 多條件篩選 | 條件是預設的，非行為驅動 |

## 7. 設計考量

### 7.1 候選池大小 vs. 約束數量

n 張卡片鎖定 k 個約束時，期望候選池大小指數下降。每個方向-值對約匹配 n/8 張卡片（假設均勻分佈）。k 個獨立鎖定下：

```
E[|P|] ≈ n / 8^k
```

n=40, k=3 時：E[|P|] ≈ 40/512 ≈ 0.08。這意味著小卡片集加多鎖定時，FIFO 解鎖會頻繁觸發。實務建議：**n 至少為 8² = 64，才能在 2 個同時鎖定下有流暢體驗**。

### 7.2 模糊匹配擴展

嚴格等式約束 `A(c, d) = v` 可放寬為容差窗口：

```
P_fuzzy(L, c_t, ε) = { c ∈ C | c ≠ c_t ∧ ∀(d, v) ∈ L : |A(c, d) - v| ≤ ε }
```

ε=1 時，每個鎖定約匹配 3/8 的卡片而非 1/8，大幅增加小資料集的候選池大小。排列約束仍然成立 — 模糊匹配只影響篩選，不影響卡片資料。

### 7.3 數值賦予策略

排列約束要求將 {1…8} 分配到每張卡片的 8 個方向。策略包括：

- **隨機排列**：簡單，均勻分佈。Prototype 中使用。
- **基於嵌入**：將卡片特徵（如透過 CLIP）映射到 8 個維度，再在每個維度內排序以產生離散值。
- **人工策展**：領域專家為小型高品質資料集手動賦值。
